# Write Help Docs Reference

This document expands on [SKILL.md](SKILL.md). Use it for input-field mapping,
MDX templates, sidebar placement, mini-examples, and anti-patterns.

## Input field glossary

Map brief headings to MDX sections:

| Brief field | MDX usage |
| ----------- | --------- |
| Suggested section | Sidebar group in `helpCenter.js` |
| Suggested slug / Target `/help/{slug}` | File path and URL |
| Change summary | Opening paragraph after frontmatter |
| User capability | `## What you can do` or woven into the intro |
| Entry points | `## How to get there` or `## Entry points` — bullet list of nav paths |
| UI labels | **Bold** in prose; exact spelling from the brief |
| Flow | `<Steps>` for how-tos; numbered list for reference pages |
| Permissions / roles | `## Who can do this` or `## Permissions` |
| Edge cases | `## Edge cases` or `<Aside type="caution">` when brief |
| Page type | Picks the template below (`how-to`, `overview`, `reference`) |
| Cross-links | `/help/{slug}` links in body text on this page and related pages |

Optional brief fields (`Source commits`, `Changed file paths`) are for traceability
only — do not paste them into user-facing help pages unless the user asks.

## File path conventions

Both layouts work in Starlight:

```
src/content/docs/help/check-in-goal.mdx          # flat
src/content/docs/help/invite-team-member/index.mdx   # folder (images colocated)
```

When using a folder, place screenshot PNGs next to `index.mdx`.

## MDX templates

### How-to (`page type: how-to`)

```mdx
---
title: "Manage company billing"
description: "Learn how to review your plan, usage, and billing actions as a company admin."
---

import { Steps, Aside } from '@astrojs/starlight/components';
import ImageEnhancer from '@/components/ImageEnhancer.astro';

<ImageEnhancer />

Brief intro from **Change summary** and **User capability**.

## How to get there

- **Company Administration** → **Manage plan**
- Company-wide danger banners → **Review billing**

![Screenshot: Billing overview with current plan and usage](./manage-company-billing-overview.png)

## Review billing and take action

<Steps>
1. Open **Billing** from Company Administration.
2. Review **Current plan**, **Usage and limits**, and **Status details**.
3. Choose an action: **Change plan**, **Update payment method**, **View billing history**, or **Review cancellation**.
4. Complete checkout or confirm cancellation, then return to the billing overview.
</Steps>

## Who can do this

Billing pages and billing CTAs require permission to manage billing
(`can_manage_billing`).

## Edge cases

<Aside type="caution">
Company-wide banners may show **Payment issue requires attention**,
**This company is read-only**, or **This company is over its plan limits**.
Collaboration may pause until billing is fixed.
</Aside>

See also [Invite a team member](/help/invite-team-member) and
[Join an organization](/help/join-organization) when plan limits affect
membership.
```

### Overview (`page type: overview`)

```mdx
---
title: "Introduction to Work Maps"
description: "Learn how Work Maps help you visualize and track work across your company or spaces."
---

import { Aside } from '@astrojs/starlight/components';
import ImageEnhancer from '@/components/ImageEnhancer.astro';

<ImageEnhancer />

Concept intro from **Change summary**.

## Feature sections

One `##` section per major capability from the brief.

## When to use this

Bullet list of scenarios (from **User capability** and **Entry points**).

<Aside>Short tip or real-time behavior note when relevant.</Aside>
```

### Reference (`page type: reference`)

```mdx
---
title: "Keyboard shortcuts"
description: "Reference of global and task keyboard shortcuts in Operately."
---

import { Aside } from '@astrojs/starlight/components';

Brief intro from **Change summary**.

## How to open the cheatsheet

- **Help** menu → **Keyboard shortcuts**
- Press **?**

## Global shortcuts

| Shortcut | Action |
| -------- | ------ |
| **Cmd+K** / **Ctrl+K** | **Open global search** |
| **?** | **Open keyboard shortcuts** |

## Task shortcuts

| Shortcut | Action |
| -------- | ------ |
| **j** / **k** | **Select next task** / **Select previous task** |
| **Return** | **Open selected task** |

<Aside>Some shortcuts only work when a task is selected or the current view supports inline editing.</Aside>

See [Global Search](/help/global-search) for search usage.
```

## Screenshots

When the brief describes a visual flow but no image file exists:

```mdx
![Screenshot: Plan limit reached modal during invite](./plan-limit-modal.png)
```

Use a descriptive alt text starting with `Screenshot:`. List all placeholder
filenames in the completion summary so a human can capture and add them.

Do not remove existing screenshots on `update_page` unless the brief says the
UI changed enough to invalidate them.

## Sidebar entry rules

Add to the `items` array inside the matching section object:

```js
{
  label: "Manage company billing",
  link: "/help/manage-company-billing",
},
```

- **Label** — readable title; align with the MDX `title` frontmatter field.
- **Link** — `/help/{slug}`; match trailing-slash style of siblings in that
  section.
- **Position** — near related pages (e.g. billing after invite/remove member
  entries in Company administration).

## Mini-examples

### `new_page` how-to — `manage-company-billing`

From brief:

- Section: Company administration
- Labels: `Billing`, `Change plan`, `Cancel plan`
- Flow: Open Billing → review → act → return
- Cross-links: invite-team-member, join-organization

Produces:

- File: `src/content/docs/help/manage-company-billing/index.mdx`
- Sidebar: Company administration section
- Cross-links on invite and join pages mentioning plan limits

### `update_page` — billing limits on `invite-team-member`

From brief:

- Target: `/help/invite-team-member`
- New labels: `Plan limit reached`, `Recommended plan`, `Next step`
- Flow: Submit invite → modal instead of success when cap hit

Produces:

- New `## Plan limits` section (or similar) in existing
  `invite-team-member/index.mdx`
- Link to `/help/manage-company-billing`
- Preserved existing invite methods and screenshots

## Product repo lookup

When the **operately** product repo is available (often as a sibling directory
such as `../operately/`), you may read source to clarify the brief. This is
optional — the brief remains the primary source of truth, and the task must
still be completable when operately is not on disk. If something is still
unclear after checking the brief (and optionally the repo), ask the user.

Where to look in the operately repo:

### Primary (user-facing UI)

| Path | What to find |
| ---- | ------------ |
| `app/assets/js/pages/` | Screens, wizards, settings, page layout |
| `app/assets/js/features/` | Feature UI, modals, forms, activities |
| `turboui/src/` | Shared components; `*Page` and task/billing UI |
| `app/assets/js/routes/paths.tsx` | Routes |
| `app/assets/js/layouts/` | Top nav, sidebar, dropdowns |
| `ee/assets/js/` | Enterprise/admin UI |

### Secondary (validation)

| Path | Use |
| ---- | --- |
| `app/test/features/` | Confirm user flows and step order |
| `specs/` | Product intent for major features |

Do not use operately backend or API paths to write manual help pages — API docs
under `src/content/docs/help/api/` are CI-generated. Use backend code only to
locate which frontend page renders a feature, then read the UI source.

## Parsing unstructured briefs

When the input is not a formal audit, infer actions:

| Signal | Likely action |
| ------ | ------------- |
| "No existing page" / new feature name | `new_page` |
| "Update" / "stale" / `/help/slug` cited | `update_page` |
| API endpoint / GraphQL / serializer | `api_auto_sync` — do not write |
| Internal refactor / tests only | `skip` |
| "Needs product decision" / "conditional" | Ask user |

Extract or ask for: slug, section, page type, UI labels, flow, permissions,
cross-links.

## Anti-patterns

**Do not paraphrase UI labels.** Write **Change plan**, not "modify your
subscription."

**Do not write API docs manually.** Items marked `api_auto_sync` are published
by CI to `src/content/docs/help/api/`. Never create or edit files there.

**Do not rely on sidebar-only linking.** New pages need body links from at
least one related existing page.

**Do not invent flows.** If the brief lacks step order, ask — do not guess from
general product knowledge.

**Do not normalize `helpCenter.js` link style globally.** Copy the pattern used
by neighboring entries in the same section.

**Do not delete unrelated content on `update_page`.** Surgical edits only.

**Do not document conditional items without confirmation.** Ask when audience
or section placement is unclear.

## Common mistakes to avoid

- Forgetting to update cross-linked pages when adding a high-priority new page
- Adding a sidebar entry but no MDX file (or the reverse)
- Using `Aside` from TurboUI or other imports — use `@astrojs/starlight/components` only
- Hard-coding full URLs — use `/help/{slug}` paths
