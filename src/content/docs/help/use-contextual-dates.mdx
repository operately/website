Contextual dates let you choose the right level of precision for a plan. Instead of forcing every deadline to be a specific day, Operately lets you set dates as a **day**, **month**, **quarter**, or **year**.

This helps teams plan work more honestly. Some commitments need an exact day. Others are better expressed as "Q2" or "September" until the details are clearer.

## When to use contextual dates

Use contextual dates whenever the timeline matters, but the exact precision should match the nature of the work.

For example:

- **Day** for a concrete deadline like a launch, review, or handoff
- **Month** for work that should finish within a broader window
- **Quarter** for roadmap items, strategic initiatives, or OKRs
- **Year** for long-term goals or multi-phase efforts

## Available date types

### Day

Choose **Day** when the exact calendar date matters.

Use it for:
- Launches
- External deadlines
- Fixed review dates
- Time-sensitive deliverables

Example: **March 18, 2026**

### Month

Choose **Month** when you know the target month, but not the exact day yet.

Use it for:
- Work planned within a monthly cycle
- Deliverables with some scheduling flexibility
- Early planning before a precise date is agreed

Example: **March 2026**

### Quarter

Choose **Quarter** when the commitment is tied to a planning cycle rather than a specific date.

Use it for:
- Quarterly goals
- Roadmap commitments
- Strategic initiatives
- Work reviewed at quarter boundaries

Example: **Q2 2026**

### Year

Choose **Year** when the work spans a long horizon and a finer date would imply false precision.

Use it for:
- Annual objectives
- Long-term transformation work
- Multi-stage initiatives still taking shape

Example: **2026**

## How to pick the right level of precision

A good rule is: **be as specific as reality allows, but no more specific than that**.

Choose the earliest date type that still feels truthful:

- If the exact day is known, use **Day**
- If only the month is known, use **Month**
- If planning is quarter-based, use **Quarter**
- If the work is long-range, use **Year**

Avoid choosing a day just because the interface allows it. Exact dates can create false certainty and unnecessary churn.

## How to set a contextual date

<Steps>
1. Open the item you want to update, such as a goal, project, or milestone
2. Click the current date field or the option to set a date
3. Choose the date type that matches your planning confidence
4. Select the value for that type
5. Confirm the change
</Steps>

## Why this is useful

Contextual dates help teams:

- Plan with less false precision
- Communicate confidence more clearly
- Align work with monthly, quarterly, or annual planning cycles
- Update timelines without overcommitting too early

They are especially useful when you are moving from strategy into execution and different pieces of work have different levels of clarity.
